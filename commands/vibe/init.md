## Development workflow (Vibe Coding)

The user is the **Product Owner only**. They describe requirements and evaluate outcomes тАФ they do NOT write or manually test code.

### Definition of Done

A task is complete ONLY when ALL of the following are true:

- [ ] Implementation code is written
- [ ] Tests covering the nominal path exist and pass
- [ ] Tests covering edge cases and error paths exist and pass
- [ ] Lint exits with code 0 тАФ style enforced by tooling (see manifest for command)
- [ ] All tests pass тАФ run test command from manifest
- [ ] No debug artifacts left in code (console.log, print, dbg!, etc.)

**Never present a result to the user if tests are failing.**

### TDD workflow

For every feature or fix:

1. **Write tests first** тАФ describe expected behavior through tests before writing implementation
2. **Confirm tests fail** тАФ run the test command and verify new tests fail (red)
3. **Write implementation** тАФ minimum code to make tests pass
4. **Confirm tests pass** тАФ run the test command and verify all tests pass (green)
5. **Refactor if needed** тАФ clean up while keeping tests green
6. **Run lint** тАФ run the lint command and fix any issues
7. **Present result** тАФ summarize what was done, what was tested, and the test output

### Bug fix workflow

1. **Reproduce in a test first** тАФ write a failing test that captures the bug
2. Fix the bug
3. Confirm the test passes
4. Present result with the test name that now covers the bug

### Self-correction loop

If tests or lint fail:
- Do NOT ask the user for help
- Diagnose the failure, fix it, re-run
- Repeat until green (max 3 self-correction attempts)
- Only escalate to the user if the failure reveals an ambiguous requirement
